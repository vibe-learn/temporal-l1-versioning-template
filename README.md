        # temporal — Версионирование workflow

        Homework-шаблон для урока **l1_versioning** (Версионирование workflow) на платформе Vibe Learn.

        ## Что делать

        Дан OrderWorkflow на Go SDK (go.temporal.io/sdk) с двумя активностями в определённом
порядке. Шаг 1: напиши replay-тест (worker.NewWorkflowReplayer) на сохранённой истории
«версии 1». Шаг 2: внеси несовместимое изменение (добавь активность в середину) и убедись,
что replay-тест падает non-determinism-ошибкой. Шаг 3: оберни изменение в
workflow.GetVersion с changeID и веткой DefaultVersion/1 так, чтобы replay старой
истории снова прошёл, а новые workflow шли по новой ветке. Тесты проверят оба сценария.

## Контекст (из transfer-задачи урока)

Workflow SubscriptionWorkflow живёт месяц: при старте списывает деньги, потом
workflow.Sleep на 30 дней, потом продлевает. В проде сейчас крутится ~20 000 таких
workflow на разных стадиях (многие спят). Продукт просит добавить проверку на фрод
ПЕРЕД списанием (новая активность FraudCheck первым шагом). Разработчик добавил вызов
FraudCheck в начало функции и собирается деплоить.

**Вопрос:** разбери ситуацию. Опиши:
(a) что произойдёт с 20 000 спящих workflow при таком деплое и почему;
(b) как сделать изменение безопасно через GetVersion (что в старой ветке, что в новой);
(c) почему дренаж тут плохой вариант, а где бы он подошёл.

## Recap из урока

- **Изменение workflow-кода ломает replay уже запущенных экземпляров**: их продолжение проигрывает старую историю новым кодом, и расхождение последовательности команд → **non-determinism error**.
- Безопасно: менять **тело активности** (не реплеится). Опасно: менять **порядок/удалять активности, вставлять таймер/сигнал в середину** — всё, что меняет последовательность команд workflow.
- **workflow.GetVersion(ctx, changeID, min, max)** ветвит старую/новую логику и пишет версию в историю маркером. Старые workflow → старая ветка (DefaultVersion), новые → новая. Оба детерминированы.
- Альтернативы: **Worker Versioning (Build IDs)** — старые воркеры дорабатывают старые workflow; **дренаж** — дождаться завершения in-flight; **ContinueAsNew** — самоперезапуск с чистой историей.
- Главная ловушка — недооценить, что считается изменением workflow-логики. Прогоняй любое изменение мысленно через replay: «совпадёт ли последовательность команд со старой историей?».

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go` (workflow + активности), прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - SDK: `go.temporal.io/sdk`
        - Docker + docker-compose — `docker compose up` поднимает Temporal dev server на `:7233` + Web UI на `:8233`. Адрес переопределяется через env `TEMPORAL_ADDRESS` (дефолт `localhost:7233`).
        - Юнит-тесты на `testsuite.TestWorkflowEnvironment` (активности замоканы) бегут в CI БЕЗ сервера; интеграционный тест включается через `TEMPORAL_INTEGRATION=1`.

        ## Запуск

        ```bash
        # Поднять локальный Temporal dev server + UI
        docker compose up -d
        # Web UI: http://localhost:8233

        # Прогнать тесты (юнит на TestWorkflowEnvironment — без сервера;
        # интеграционный включается через TEMPORAL_INTEGRATION=1)
        go test ./...
        TEMPORAL_INTEGRATION=1 go test ./...

        # Запустить воркер (регистрирует workflow + активности, слушает task queue)
        go run .
        ```
